<objective>
Add pane-ready detection to tmux session Start() so that the tool command is only sent after the pane shell is initialized and accepting input. This fixes the WSL/Linux timing race where SendKeysAndEnter fires before the pane shell prompt is ready, causing tool processes to silently fail.

Purpose: Resolves PLAT-01 (#311) by ensuring the tmux pane has a ready shell before dispatching the tool launch command. Without this, non-interactive invocations on WSL/Linux can hit a race condition where keystrokes are swallowed by the pane initialization sequence.

Output: `internal/tmux/pane_ready.go` (new file), modified `internal/tmux/tmux.go`, and `internal/tmux/pane_ready_test.go` (new test file).
</objective>

    ```go
    // Wait for pane shell to be ready before sending command via send-keys.
    // On WSL/Linux non-interactive contexts, tmux pane initialization can take
    // 100-500ms after new-session -d. Without this wait, send-keys may fire
    // before the shell prompt is ready, causing the command to be silently lost.
    // This is a no-op when the pane is already ready (typical on macOS).
    if command != "" && !startWithInitialProcess {
        paneReadyTimeout := 2 * time.Second
        if platform.IsWSL() {
            // WSL2 cold-start (no prior tmux server) can take 3-4s
            paneReadyTimeout = 5 * time.Second
        }
        if err := waitForPaneReady(s, paneReadyTimeout); err != nil {
            statusLog.Warn("pane_ready_timeout",
                slog.String("session", s.Name),
                slog.String("timeout", paneReadyTimeout.String()),
                slog.String("error", err.Error()),
            )
            // Non-fatal: proceed with SendKeysAndEnter anyway.
            // On fast machines the pane is likely ready despite the timeout.
        }
    }
    ```
